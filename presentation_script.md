## 🟡 왜 Virtual DOM이 필요한가 (0:40 ~ 1:20)

[ 패널 1 — Vanilla 버전 피드 보여주기 ]

"먼저 질문 하나 드릴게요.

Instagram 좋아요를 누르면 브라우저 내부에서 무슨 일이 일어날까요?

💬 DOM 설명
브라우저는 HTML을 읽어서 DOM이라는 트리 구조를 만들어요.
화면에 보이는 모든 요소가 이 트리의 노드예요.

문제는 이 DOM을 건드릴 때마다
브라우저가 전체 레이아웃을 다시 계산하고 (Reflow)
화면을 다시 그려야 해요 (Repaint)

좋아요 숫자 하나만 바꿔도
이 과정이 전체적으로 일어나요.

💬 Vanilla의 한계
Vanilla JS로 만든 A버전을 보시면
좋아요를 누를 때마다 해당 포스트 전체를 다시 그려요.
1000번 누르면 1000번 전체를 다시 그립니다.

그래서 React는 Virtual DOM을 도입했어요."

## 🏁 인피니티 스크롤

[ 패널 1 — 인피니티 스크롤 시연 ]

"아까 말씀드린 인피니티 스크롤입니다.

💬 센티널 개념
스크롤 감지는 Intersection Observer API를 써요. 
이는 사실 React와 관련 없는 라이브러리 입니다.
피드 맨 아래에 보이지 않는 div를 하나 놔둬요.
이걸 센티널, 파수꾼이라고 해요.

사용자가 스크롤해서 이 센티널이 화면에 들어오는 순간
새 포스트를 로드해요.

scroll 이벤트처럼 매 픽셀마다 감지하는 게 아니라
보임 / 안 보임 두 가지 상태만 감지하기 때문에
훨씬 효율적이에요."

[ 피드 스크롤 → 포스트 추가 확인 ]

"포스트가 추가될 때
저희는 key 기반 VDom을 구현했어요.
각 포스트에 고유한 key가 있으면
Diff가 인덱스가 아니라 key로 노드를 식별해요.

새 포스트가 추가되거나 순서가 바뀌어도
key가 같으면 동일한 노드로 인식해서
기존 포스트는 건드리지 않고
새 포스트만 appendChild로 추가해요.

key 기반 Diff 덕분이에요.
각 포스트에 key가 있으면
순서가 바뀌거나 추가돼도
동일한 포스트는 그대로 두고
새 포스트만 appendChild로 추가합니다."

## 🔴 Virtual DOM + Diff 설명 (1:20 ~ 2:10)

[ 패널 2 — VDom Inspector 클릭 ]

"💬 Virtual DOM이란
실제 DOM의 가벼운 복사본이에요.
JS 객체로 만들어져 있어서 수정이 빠릅니다.

흐름은 이래요.

좋아요를 누르면
→ 실제 DOM을 바로 건드리는 게 아니라
→ 먼저 Virtual DOM에 변경사항을 반영하고
→ 이전 Virtual DOM과 새 Virtual DOM을 비교해요

이 비교 과정이 Diff 알고리즘입니다."

[ 좋아요 클릭 후 VDom Inspector 확인 ]

"보시면 왼쪽이 이전 VNode 트리, 오른쪽이 현재 VNode 트리예요.

전체 노드 중에 노란색으로 표시된 이 노드 하나만 바뀌었어요.
span.likes 안의 텍스트 — 127이 128로.

💬 Diff 알고리즘 5케이스
Diff는 두 트리를 위에서부터 순서대로 비교하면서
다섯 가지 케이스로 분류해요.

노드가 새로 생겼다   → 추가 (Add)
노드가 사라졌다     → 삭제 (Delete)
태그 종류가 바뀌었다 → 통째로 교체 (Props)
텍스트가 바뀌었다   → 텍스트만 수정 (Update)
속성이 바뀌었다     → 속성만 수정 (Update)

완전 탐색하면 O(n³)이지만
이 다섯 규칙으로 O(n)으로 줄일 수 있어요.

지금 좋아요 케이스는 네 번째, 텍스트 변경이에요."

💬 Fiber — 왜 필요한가
"그런데 Diff가 비교할 노드가 수백, 수천 개라면 어떨까요?

한 번에 다 비교하면 그동안 브라우저가 멈춰요.
스크롤도 안 되고, 버튼도 안 눌려요.

그래서 React는 Fiber라는 구조를 만들었어요.
Diff 작업을 한 번에 하는 게 아니라
작은 조각으로 쪼개서 조금씩 처리해요.

노드 10개 비교하고 → 브라우저한테 양보 (클릭 처리)
다음 10개 비교하고 → 브라우저한테 양보 (스크롤 처리)
남은 노드 비교하고 → 변경분만 DOM에 반영

이러면 렌더링 중에도 사용자가 클릭하거나 스크롤할 수 있어요.
이게 인스타그램이 수백 개 포스트를 로드하면서도
부드럽게 동작하는 이유입니다.

뒤에 벤치마크에서 이 차이를 직접 수치로 보여드릴게요."

## 🟣 Diff & Patch 시연 (2:10 ~ 3:10)

[ 패널 3 클릭 ]

"이게 이번 과제의 핵심 요구사항입니다.

왼쪽이 실제 영역, 오른쪽이 테스트 영역이에요.

페이지를 로드하면 실제 영역의 HTML을
Virtual DOM 트리로 변환하고
그 Virtual DOM으로 테스트 영역을 렌더링합니다.

💬 Props 설명
테스트 영역에서 HTML을 수정할 수 있는데
태그 속성을 바꾸면 Props 변경 케이스가 동작해요.

Props는 태그에 붙어있는 모든 속성이에요.
class, id, style, onclick 전부 Props입니다.

VNode 안에 해시테이블처럼 저장됩니다.
props: { class: 'like-btn', id: 'btn-1' }

Diff는 이전 props와 새 props를 비교해서
바뀐 속성만 골라내고
Patch가 그 속성만 setAttribute로 반영해요."

[ 테스트 영역에서 좋아요 숫자 127 → 999로 수정 ]

"127을 999로 바꿔볼게요. Patch를 눌러보겠습니다."

[ Patch 클릭 ]

"Diff 결과를 보시면
🟡 UPDATE — span.likes — 127에서 999로.
1개 변경 감지, DOM 1회 업데이트.

실제 영역에 숫자만 바뀌었어요.
나머지 노드는 건드리지 않았습니다.

💬 Patch란
Diff가 차이점을 찾는 탐색 과정이라면,
Patch는 적용시키는 부분입니다.

(Diff가 탐정이라면 Patch는 시공업자예요.)
Diff가 '여기 바꿔야 해' 목록을 만들면
Patch가 그 목록대로 실제 DOM에 반영해요.

추가는 appendChild
삭제는 removeChild
교체는 replaceChild
텍스트는 textContent 수정
속성은 setAttribute, removeAttribute"

[ 태그를 p에서 span으로 변경 후 Patch ]

"태그 종류를 바꾸면 어떻게 될까요.
p를 span으로 바꾸고 Patch를 눌러볼게요.

🔴 DELETE p
🟢 ADD span

타입이 다르면 수정이 아니라 통째로 교체해요.
자식 노드까지 전부 새로 만들어요.
이게 타입 변경 케이스입니다."

### 1. 좋아요 1000회 연속

"가장 흔한 인터랙션이에요.
좋아요를 1000번 연속으로 누르면 어떻게 될까요.

Vanilla는 매번 innerHTML로 전체를 다시 그려요.
Mini React는 VNode 객체만 비교해요.
Real React는 배치로 한 번에 처리해요.

💬 Mini React가 Vanilla보다 느릴 수도 있어요.
벤치마크가 단순한 DOM으로 시뮬레이션하기 때문이에요.

💬 속도 차이보다 중요한 건

### 4. UI 블로킹 (응답 지연 시간)

"이 테스트는 브라우저가 얼마나 멈추는지를 측정해요.

원리는 간단합니다.
브라우저한테 '이 일이 끝나면 알려줘'라고 미리 부탁해놓고
무거운 작업을 시작하는 거예요.

작업이 가벼우면 금방 '끝났어요' 알림이 와요.
작업이 무거우면 한참 뒤에야 알림이 와요.

이 알림이 올 때까지 걸린 시간이
곧 사용자가 버튼을 눌러도 반응 없는 시간이에요.

Vanilla는 DOM 조작 1000회로 수백ms 지연.
Mini React는 JS 연산만이라 빨라요.

💬 Real React가 Mini React보다 느릴 수 있어요.
불변성 기반으로 map()을 1000번 돌리면서
매번 새 배열과 새 객체를 만들기 때문이에요.
이건 불변성의 트레이드오프입니다.
실제 앱에서는 이런 극단적 반복이 없으니 문제없어요.

💬 Fiber가 해결하는 방법
Fiber는 작업을 작은 단위로 쪼개서
브라우저한테 양보하면서 처리해요.
requestIdleCallback으로 한가할 때만 작업해요.

인스타그램이 수백 개 포스트를 로드하면서도
스크롤이 부드러운 이유가 바로 이겁니다."

### 5. setState 3회 동시 호출 (Batching)

"마지막은 Batch입니다.

Vanilla는 setState 3번 → 렌더링 3번.
Mini React와 Real React는 3번 호출해도 렌더링 1번.

💬 useState란
함수가 실행될 때마다 변수가 사라지지만
useState는 Fiber의 hooks[] 배열에 저장해서 기억해요.

💬 Batch와 Fiber의 연결
Fiber의 업데이트 큐에 모아뒀다가
workLoop가 실행될 때 한 번에 처리해요.
이게 React 18의 Automatic Batching과 같은 원리입니다."

모든 게 하나의 흐름으로 연결되어 있어요.

인스타그램 좋아요 버튼 하나가 이렇게 동작합니다.

이상입니다. 감사합니다."

## ❓ 예상 QnA

**Q. Vanilla에서 인피니티 스크롤을 구현하지 않은 이유가 있나요?**
```
Vanilla에서도 인피니티 스크롤 자체는 구현할 수 있어요.
Intersection Observer는 React와 무관한 브라우저 API거든요.

차이는 포스트가 추가될 때 일어나요.

Vanilla는 새 포스트 10개를 추가할 때
전체를 다시 렌더링해요.

Mini React는 Diff가 새 포스트 10개만 감지해서
기존 포스트는 건드리지 않고
새 것만 appendChild로 추가해요.

이게 key 기반 Diff의 효과입니다.
```

**Q. Vanilla가 느린 정확한 이유가 뭔가요?**
```
DOM을 건드릴 때마다 브라우저가
Reflow와 Repaint를 전체적으로 수행하기 때문이에요.

Reflow는 레이아웃을 다시 계산하는 거고
Repaint는 화면을 다시 그리는 거예요.

텍스트 하나 바꿔도 이 과정이 일어나요.
벤치마크에서 측정한 수치로 직접 보여드릴 수 있어요.
```

**Q. Fiber 없이도 Diff와 Patch는 동작하나요?**
```
네, 동작해요.
저희 v1~v3가 Fiber 없이 동작하는 버전이에요.

다만 동기 방식이라서
노드가 많아질수록 브라우저가 멈추는 현상이 생겨요.

Fiber는 그 문제를 해결하기 위해 추가된 거예요.
Diff의 결과를 비동기로 처리할 수 있게 해줘요.
```

**Q. useState 순서가 바뀌면 어떻게 되나요?**
```
hooks 배열의 인덱스로 상태를 구분하기 때문에
순서가 바뀌면 엉뚱한 상태를 읽어버려요.

예를 들어
첫 번째 useState가 count
두 번째 useState가 liked 라면

조건문으로 첫 번째를 건너뛰면
liked가 hooks[0]을 읽어버려요.

React가 훅을 조건문 안에 쓰면 안 된다는 규칙의
정확한 이유입니다.
```

**Q. Mini React와 Real React 성능이 비슷한 이유는?**
```
구현 대응표에서 보시듯
핵심 원리가 동일하기 때문이에요.

차이는 저희가 처리하지 못한 엣지케이스 수준에 있어요.
Real React는 수천 개의 엣지케이스를 처리하고
메모이제이션, Error Boundary 같은 추가 최적화가 있어요.

하지만 핵심 흐름인
Virtual DOM → Diff → Fiber → Batch는 동일해요.
```

**Q. XSS 공격은 어떻게 막았나요?**
```
댓글 입력 시 innerHTML 대신 textContent를 사용했어요.

innerHTML은 태그를 그대로 실행하지만
textContent는 전부 텍스트로만 표시해요.

script 태그를 입력해도 실행되지 않아요.

Virtual DOM을 쓰면
사용자 입력을 항상 값으로만 처리하기 때문에
이런 보안 이점이 자연스럽게 생겨요.
```
